FarmPulse AI Website: Complete Feature and Flow Explainer

This file is a plain-text explanation of the full FarmPulse AI website so that a human reviewer, judge, crawler, or non-JavaScript fetcher can understand exactly what exists in the product, how many pages there are, what every page does, and how the system works underneath.

==================================================
1. WEBSITE OVERVIEW
==================================================

FarmPulse AI is a multi-page React single-page application backed by a FastAPI API. It is built as a hackathon-ready crop-risk intelligence platform for Indian agriculture.

The website currently has 7 main user-facing pages:

1. /overview
2. /farmer-advisory
3. /channel-fallback
4. /districts
5. /mandi-prices
6. /institutional
7. /audit

There is also a root route / which automatically redirects to /overview.

The website is designed around one central product promise:
- a farmer or institutional user can get district-aware crop-risk intelligence, multilingual advisory, visible agent reasoning, and auditable outputs in one connected flow.

==================================================
2. GLOBAL APP STRUCTURE
==================================================

The app uses a shared shell and navigation system.

2.1 Shared layout
- The app uses a full-height layout with a sidebar on large screens.
- On smaller screens, the sidebar is replaced with a sticky horizontal mobile navigation row.
- The main content area scrolls internally, not the full page shell.
- Each page is wrapped inside a max-width content container.

2.2 Sidebar navigation items
The left sidebar contains these entries:
- Dashboard -> /overview
- Farmer Help -> /farmer-advisory
- Channels -> /channel-fallback
- District Data -> /districts
- Mandi Prices -> /mandi-prices
- Institutional Reports -> /institutional

2.3 Sidebar utility content
The sidebar also contains:
- a FarmPulse AI identity block with ET GenAI 2026 branding
- a small “Simple use” helper card telling users to start with Farmer Help and then explore Channels and Mandi Prices
- a theme toggle button for dark/light mode
- an “Audit On” badge indicating auditability is active

2.4 Shared page header pattern
Every page uses a common header component with:
- eyebrow label

- descriptive summary text
- optional page-level action button

==================================================
4. FRONTEND API COMMUNICATION
==================================================

The frontend calls the backend through a shared API client layer.

Important frontend API functions:
- fetchDistricts() -> GET /api/districts
- fetchDistrictDetail(districtId) -> GET /api/district/{districtId}
- fetchMandiPrice({ district, crop }) -> GET /api/mandi-price
- runAnalysis(payload) -> POST /api/analyze
- runFarmerQuery(payload) -> POST /api/farmer-query
- simulateEdgeCase(payload) -> POST /api/simulate-edge-case
- fetchAuditLog(filters) -> GET /api/audit-log
- createEventSource(runId) -> GET /api/stream/{runId} via EventSource
- exportAuditLogUrl() -> GET /api/audit-log/export

4.1 Audit cache behavior
The frontend also stores audit entries in browser localStorage.
This is important because:
- the backend uses SQLite
- serverless environments can be ephemeral
- the app merges backend audit entries with locally cached entries
- this keeps the audit page useful during demos even when persistence is imperfect

4.2 API base URL
The frontend expects a Vercel build variable:
- VITE_API_BASE=https://api-orcin-alpha.vercel.app

7. BACKEND ARCHITECTURE AND HOW THE WEBSITE WORKS
==================================================

The backend is a FastAPI application called FarmPulse Agent Engine.

7.1 Main API endpoints
- GET /health
- POST /api/analyze
- POST /api/farmer-query
- POST /api/simulate-edge-case
- GET /api/districts
- GET /api/district/{district_id}
- GET /api/mandi-price
- GET /api/audit-log
- GET /api/audit-log/export
- GET /api/stream/{run_id}

7.2 Core 4-agent workflow
The main execution flow is run_pipeline().
It executes these stages in order:
1. satellite_scout
2. crop_risk_analyst
3. advisory_generator
4. institutional_reporter
5. human_escalation_node if escalation is required

7.3 Streaming behavior
The backend keeps a per-run asyncio queue.
As each stage runs, it emits SSE events via emit_event().
The frontend listens to these events on /api/stream/{run_id}.

This powers:
- the live activity feed on /overview
- the live 4-agent step cards on /farmer-advisory

8. BACKEND DATA SOURCES AND LOGIC
==================================================

8.1 District records
The backend has a built-in list of agricultural districts with:
- id
- district name
- state
- coordinates
- primary crop
- acreage
- KVK contact
- default language

Covered states include:
- Maharashtra
- Punjab
- UP
- MP
- Rajasthan

8.2 Language handling
The backend supports these languages:
- Auto
- English
- Hindi
- Marathi
- Punjabi
- Bengali
- Gujarati
- Tamil
- Telugu
- Kannada
- Malayalam

Language resolution logic:
- normalize manual language labels
- if Auto, inspect the query text by Unicode range/script
- use district default language to disambiguate Devanagari where needed

8.3 Weather data
The backend fetches forecast data from Open-Meteo.
It calculates:
- 7-day rainfall
- rainfall anomaly percent
- average temperature
- temperature anomaly
- average humidity
- weather anomaly type

Weather anomaly types include:
- normal
- flood
- drought
- heat
- cold

If weather fetch fails:
- retries up to 3 times
- falls back to last-known-good weather if available
- otherwise generates a synthetic fallback weather block marked stale

8.4 Vegetation signal
The current deployed product does not use pixel-level district satellite extraction.
Instead it uses a reference_snapshot vegetation signal.

How that works:
- fetch NASA POWER daily agroclimate observations for the district point
- use rainfall, temperature, humidity, and solar data
- convert those conditions into a vegetation proxy anchored to MODIS reference ranges
- return NDVI-like values and provenance metadata

Modes:
- reference_snapshot when NASA POWER succeeds
- demo_estimate when live observations are unavailable

The UI is intentionally transparent about this.

8.5 Crop calendar
The backend contains crop calendars by state and crop.
These provide:
- stage_by_month
- days_to_harvest

This is used in risk reasoning and PHI-style compliance logic.

8.6 Pest windows
The backend contains hardcoded pest windows for some district/crop/month combinations.
Examples include:
- Pink bollworm in Yavatmal cotton
- Whitefly in Bathinda cotton
- Stem fly in Indore soybean
- Aphid in Ludhiana wheat

8.7 Soil snapshot
The soil snapshot is generated from:
- weather conditions
- vegetation score
- seeded district randomness

It returns:
- moisture percentage
- moisture band
- soil pH
- nitrogen kg/ha
- electrical conductivity
- a summary string

8.8 5-day forecast block
The backend builds a 5-day advisory-oriented forecast with:
- day
- date
- rainMm
- maxTempC
- minTempC
- humidityPct
- outlook

This is used especially on the farmer page to choose the best fertilizer window.

8.9 Mandi pricing
The backend first tries live Agmarknet via data.gov.in if AGMARKNET_API_KEY exists.
If not, it falls back to district/crop aligned sample mandi data.

The response includes:
- modal, min, and max prices
- benchmark price
- recommendation
- recommendationShort
- sourceName
- sourceUrl
- live boolean
- note

==================================================
9. AGENT STAGE DETAILS
==================================================

9.1 Satellite Stress Scout
Responsibilities:
- resolve district
- fetch vegetation observation
- compute NDVI anomaly against seasonal baseline
- compute confidence
- detect staleness
- fetch weather
- produce satellite signal summary
- add initial reasoning chain step
- add audit entry
- emit SSE start/complete events

Important behavior:
- if freshnessDays > 10, confidence is reduced to 41%
- if NDVI anomaly is severe and rainfall is extremely high, a warning about water stress vs flood stress ambiguity is added
- in data_staleness edge case, freshness is forced older

9.2 Crop Risk Analyst
Responsibilities:
- determine crop stage and days to harvest
- determine pest probability and pest name
- compute risk score and category
- decide root cause
- decide escalate or not
- enforce unsupported crop behavior
- enforce stale-data escalation
- adjust for multi-stressor conflict
- build soil snapshot
- build 5-day forecast
- build query-aware recommendation plan
- extend reasoning chain
- add audit entry
- emit SSE start/complete events

Important guardrail behavior:
- unsupported crop -> escalate with zero confidence
- stale data -> escalate
- confidence < 60 -> escalate
- if risk is high/critical and within 21 days of harvest, chemical action is replaced by non-chemical options only

9.3 Advisory Generator
Responsibilities:
- select farmer and institutional model route identifiers
- render SMS advisory
- render WhatsApp advisory
- render institutional report
- append guardrail disclaimer for high/critical risk in institutional text
- add two audit entries
- emit SSE start/complete events

Important note:
- the backend stores internal model identifiers such as farmer-advisory-engine and institutional-analysis-engine
- the UI translates them into Haiku and Sonnet labels in the farmer summary section

9.4 Institutional Reporter
Responsibilities:
- aggregate district and state risk outputs
- build insurer rows
- build government early warning block
- detect state emergency alert threshold
